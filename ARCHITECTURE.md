# Growth Atlas 第一版软件架构

## 1. 文档目的

本文档基于 `README.md`、`MVP_SPEC.md`、`USER_FLOW.md`、`AI_CONVERSATION.md` 与 `DATA_MODEL.md`，描述 Growth Atlas V1.0 的软件架构、模块职责、系统协作方式、核心数据流和未来扩展方向。

本文档只描述架构设计，不包含代码、接口字段或数据库建表语句。

## 2. 架构目标与原则

Growth Atlas 第一版需要可靠支撑以下核心闭环：

**反思（Reflection）→ 洞察（Insight）→ 决策（Decision）→ 成长（Growth）**

架构设计遵循以下原则：

1. **先保证 MVP 闭环完整。** 优先支持个人档案、结构化 AI 对话、决策报告、成长记录和阶段性成长报告，不为暂不开发的功能提前引入复杂系统。
2. **业务规则由后端统一控制。** 前端不直接调用 OpenAI API，也不自行决定对话阶段、长期记忆写入或成长结论。
3. **AI 负责生成与归纳，系统负责约束与验证。** AI 输出必须经过结构校验、可信度标记、安全检查和用户确认，不能直接成为长期事实。
4. **用户确认优先。** Personal Profile、Reflection、Memory 和最终行动方案的正式写入均遵守各自的确认规则。
5. **最少保存与可追溯。** 正式记录优先保存结构化结果；结论应能追溯到来源对话、用户档案或成长记录。
6. **隐私与安全内建。** 身份隔离、访问控制、敏感内容最少保存和安全分支属于基础架构能力，而不是后续补丁。
7. **从模块化单体开始。** V1.0 使用边界清晰的单体 FastAPI 应用，降低开发和运维成本；业务增长后再按模块拆分。

## 3. 整体技术架构

第一版采用四层架构：

| 层级 | 技术 | 主要职责 |
| --- | --- | --- |
| 用户体验层 | React | 页面呈现、用户输入、对话交互、报告确认、历史记录与成长地图展示 |
| 应用与业务层 | FastAPI | 身份校验、业务流程、对话状态机、AI 编排、安全规则、数据读写与权限控制 |
| 数据与身份层 | Supabase | 用户认证、PostgreSQL 数据存储、行级访问控制、数据生命周期管理 |
| AI 能力层 | OpenAI API | 结构化追问、内容分析、报告生成、记忆建议与阶段性成长总结 |

系统逻辑关系如下：

**React 客户端 ⇄ FastAPI 后端 ⇄ Supabase**

**FastAPI 后端 ⇄ OpenAI API**

React 只与 FastAPI 和 Supabase 的认证流程协作。所有涉及用户成长数据的业务读写、AI 上下文选择和 OpenAI API 调用均由 FastAPI 统一处理。OpenAI API 不直接访问 Supabase，也不持有 Growth Atlas 的数据库权限。

### 3.1 部署形态

V1.0 建议采用独立部署的 React Web 应用与 FastAPI API 服务，Supabase 使用托管服务，OpenAI 使用外部 API。所有外部通信使用加密连接。

后端在无状态模式下运行：单次请求所需的持久状态来自 Supabase，因此后续可以通过增加 FastAPI 实例进行水平扩展。对话进度、用户确认状态和报告状态不能只保存在某个后端进程内。

## 4. 各层如何协作

### 4.1 React 前端

React 是用户体验层，负责：

- 注册、登录和会话状态展示。
- 首次引导、隐私与产品边界确认。
- Personal Growth Profile 的采集、预览、修改和确认。
- 困惑输入、逐轮对话、当前分析阶段和中断恢复。
- 决策报告展示、用户修正和行动方案确认。
- 用户主动保存 Reflection，并补充后续行动结果。
- 历史成长记录、Memory 管理和六维成长报告展示。
- 清楚区分用户确认内容、AI 建议、未知信息和待验证推测。

前端可以维护当前页面和输入草稿等临时界面状态，但服务端保存的对话阶段、确认状态和正式数据是最终可信来源。前端不得保存 OpenAI API 密钥，也不得绕过后端直接生成或写入正式成长结论。

### 4.2 FastAPI 后端

FastAPI 是系统的可信业务边界，负责：

- 验证 Supabase 签发的用户身份，并确保每次操作只访问当前用户的数据。
- 对外提供统一的业务 API，隔离前端、数据库和 AI 服务的实现差异。
- 驱动首次建档、日常反思、后续回顾等不同会话流程。
- 维护“聚焦问题—事实—情绪—认知—决策—成长—报告确认”的对话状态机。
- 从已确认 Profile、可用 Memory 和当前 Conversation 中选择最少且相关的 AI 上下文。
- 构建受版本管理的系统指令和任务指令，调用 OpenAI API。
- 校验 AI 返回内容的结构、阶段、来源类型和信息完整性。
- 执行安全边界、专业领域边界和异常降级策略。
- 在用户确认后，将报告、行动方案和成长记录写入 Supabase。
- 记录必要的审计信息、错误和性能指标，但避免把敏感正文写入普通日志。

FastAPI 在第一版保持模块化单体：各业务模块共享一个部署单元和数据库，但通过清晰的领域边界协作，避免路由、AI 提示和数据访问逻辑混杂。

### 4.3 Supabase

Supabase 为第一版提供两类核心能力：

1. **身份认证。** 负责账号注册、登录、令牌签发和会话管理；FastAPI 验证令牌并取得用户标识。
2. **PostgreSQL 数据存储。** 保存 User、Personal Profile、Conversation、Reflection、Growth Score、Memory 以及它们的状态、版本和来源关系。

数据库需要使用行级访问控制，所有用户数据都以用户标识隔离。服务端高权限凭据只存在于 FastAPI 的安全运行环境，绝不下发到浏览器。对于正式成长结论，应保存来源引用、确认状态和版本信息，以支持追溯、修正和删除。

Supabase 只负责可靠存储和访问控制，不负责决定 AI 应问什么、哪些推测可被采信或何时生成成长报告；这些属于 FastAPI 业务层职责。

### 4.4 OpenAI API

OpenAI API 是受控的 AI 能力提供方，负责：

- 理解用户当前输入并生成下一条结构化追问。
- 区分并归纳事实、解释、推测、未知信息、情绪与诉求。
- 发现可能的隐藏假设，提出替代视角。
- 梳理方案的收益、代价、风险、可逆性和所需条件。
- 生成决策报告草稿、Memory 建议和阶段性成长报告草稿。

OpenAI API 不负责：

- 用户认证和访问授权。
- 决定数据是否长期保存。
- 直接写入或修改 Profile、Reflection、Memory 或 Growth Score。
- 在没有用户确认和可追溯证据时形成永久用户特征。
- 替代心理、医疗、法律或投资专业人士作出结论。

后端仅发送完成当前任务所需的最少上下文，并在发送前过滤无关字段。AI 输出首先视为候选结果，必须由系统校验；涉及档案、记忆、行动和成长结论时，还必须遵守对应的用户确认流程。

## 5. 后端业务模块

### 5.1 身份与权限模块

连接 Supabase Auth，验证访问令牌，取得当前用户身份，并为所有业务操作建立数据归属边界。该模块还负责账号状态、隐私协议版本和删除请求的协调。

### 5.2 用户与偏好模块

管理 User 的基础设置、隐私确认、对话保存偏好和长期记忆总开关。它不保存详细成长画像，避免身份信息与敏感成长内容过度耦合。

### 5.3 个人档案模块

管理 Personal Profile 的草稿、待确认、已确认和历史版本。只有用户确认的档案内容可进入后续 AI 个性化上下文；AI 可以提出修改建议，但不能自动覆盖档案。

### 5.4 对话模块

管理 Conversation、消息顺序、会话类型、当前阶段、保存策略和恢复能力。它把一次复杂 AI 交流转化为可控制的状态流程，并保证一次反思聚焦一个核心问题。

Conversation 是过程数据，不等同于 Reflection，也不自动成为长期 Memory。用户可以保存结构化成长记录，而不永久保留完整逐字对话。

### 5.5 AI 编排模块

AI 编排模块是 FastAPI 与 OpenAI API 之间的适配层，负责：

- 根据会话类型和当前阶段选择提示模板。
- 组合当前消息、已确认档案、已授权 Memory 和必要的历史摘要。
- 控制上下文长度和隐私最小化。
- 请求结构化输出并进行字段、类型和完整性校验。
- 记录模型、提示模板和输出结构的版本，保证结果可追溯。
- 处理超时、限流、无效输出和安全拒答。

业务模块不直接散落调用 OpenAI API，而统一经过此模块，以便未来更换模型、调整提示策略和控制成本。

### 5.6 安全与内容边界模块

在普通成长分析之前和 AI 输出之后执行安全检查。若发现自伤、伤人或其他紧急危险信号，系统中止常规成长流程，切换到安全支持分支，提示用户联系当地紧急服务和可信赖的真人。

对于医疗、法律和投资等高风险问题，该模块限制输出为信息整理、问题清单和一般性权衡，不允许生成替代专业人士的确定性结论。安全事件只保存支持当下流程所需的最少信息。

### 5.7 报告模块

负责把已经完成的五层分析整理为决策报告候选稿，包括核心问题、事实与未知、情绪与诉求、认知挑战、方案权衡、AI 建议、待确认行动和回顾重点。

报告首先处于待确认状态。用户可以修改误解，并确认、调整或自行填写行动方案；未经确认的报告不能作为正式成长记录中的确定事实。

### 5.8 成长记录模块

管理 Reflection 的创建、保存、回顾、归档和删除。只有用户主动选择保存后，报告结果才成为正式 Reflection。后续行动结果可追加到原记录，并保留更新与来源关系。

### 5.9 长期记忆模块

管理 Memory 的建议、展示、确认、适用范围、有效期、停用和删除。只有用户明确同意的 Memory 才能跨对话调用；AI 推测或普通 Conversation 内容不能自动成为 Memory。

该模块还负责记录 Memory 的来源和使用情况。来源被删除或修改后，关联 Memory 需要被标记为待复核。

### 5.10 成长评估模块

基于用户确认的 Profile、已保存 Reflection 和行动回顾生成六维阶段性 Growth Score。第一版优先使用“状态 + 趋势 + 证据”，而不是伪精确数字。

每项判断必须引用证据并说明信息缺口。证据不足时输出“信息不足”或“初步观察”；用户否定或修改结论后，旧结论不能继续被当作确定事实。

### 5.11 数据访问与审计模块

统一封装对 Supabase 的读写、事务、用户隔离、软状态变化和实际删除流程。审计记录关注谁在何时对哪类数据执行了什么操作，不在普通运行日志中复制敏感对话正文。

## 6. 核心数据设计与关系

第一版沿用 `DATA_MODEL.md` 定义的六类核心数据：

| 数据实体 | 系统角色 | 关键关系 |
| --- | --- | --- |
| User | 身份与偏好的归属起点 | 拥有其他所有个人数据 |
| Personal Profile | 经确认的长期背景 | 一个用户当前最多一份有效档案，可保留历史版本 |
| Conversation | 结构化 AI 对话过程 | 一个用户可有多个；一次主对话最多形成一条主要 Reflection |
| Reflection | 用户主动保存的成长成果 | 可引用 Conversation、Profile 和 Memory，可追加行动回顾 |
| Growth Score | 六维阶段性总结 | 引用 Profile、Reflection 与回顾作为证据 |
| Memory | 经单独同意的跨对话信息 | 来源于 Profile、Reflection 或用户直接创建 |

为保持可信度，数据至少需要表达：所属用户、状态、来源、用户确认状态、创建与更新时间、版本或有效期、删除状态。AI 推测和用户确认内容必须可区分。

需要原子完成的写入应使用数据库事务。例如，用户确认保存成长记录时，Reflection 主体、用户最终行动、来源关联和 Conversation 状态应作为同一业务操作完成，避免部分成功导致数据不一致。

## 7. 一次完整用户请求的数据流

以下流程描述用户输入一个问题，到 AI 分析、生成报告并保存成长记录的完整链路。

### 7.1 用户输入问题

1. React 获取用户输入，并携带 Supabase 登录令牌向 FastAPI 发起请求。
2. FastAPI 验证用户身份、隐私状态、账号状态和请求归属。
3. 对话模块创建或恢复 Conversation，保存用户消息，并把会话置于“聚焦问题”阶段。
4. 安全模块检查输入是否触发紧急危险或高风险专业边界。若触发，则进入相应分支，不继续普通成长分析。

### 7.2 AI 分析与结构化追问

1. FastAPI 读取当前 Conversation、已确认 Personal Profile，以及用户允许且与当前主题相关的 Memory。
2. AI 编排模块只选择当前任务必需的上下文，标明信息来源和可信度，并构建本阶段的请求。
3. FastAPI 调用 OpenAI API。OpenAI 返回对问题的理解、当前阶段提取结果以及下一条追问候选。
4. 后端校验 AI 输出结构，并再次检查安全边界；有效结果被保存到 Conversation，React 展示给用户。
5. 用户继续回答，系统依次推进事实、情绪、认知、决策和成长五个阶段。每一阶段的关键总结均允许用户纠正。
6. 如果信息不足，系统继续追问或将字段标记为未知；不得由 AI 猜测补齐事实。

### 7.3 生成报告

1. 当核心问题已聚焦，五层信息达到最低充分条件，且没有未处理的安全分支时，对话模块允许进入“报告生成”阶段。
2. AI 编排模块把已确认事实、未知信息、情绪与诉求、关键假设、方案权衡和行动约束发送给 OpenAI API。
3. OpenAI 生成结构化决策报告候选稿。
4. FastAPI 校验报告必需部分、来源标记、信息可信度和行动建议边界，并将报告保存为待确认状态。
5. React 展示报告。用户可以修改误解，接受、调整或自行填写下一步行动。
6. 用户确认后，FastAPI 保存最终报告和用户确认的行动方案；AI 建议与用户最终决定保持区分。

### 7.4 保存成长记录

1. React 明确询问用户是否保存为成长记录；报告确认不等于自动同意长期保存。
2. 用户选择保存后，FastAPI 检查报告状态和数据归属。
3. 成长记录模块从已确认报告生成正式 Reflection，保存问题摘要、关键分析、个人洞察、用户行动及来源引用，并关联原 Conversation。
4. 数据访问模块通过事务完成 Reflection 与关联状态写入，Supabase 返回成功结果。
5. React 展示保存成功的记录。完整 Conversation 是否长期保留，按用户独立的保存选择处理。
6. 用户行动后可回到 Reflection 补充结果与变化；这些已确认回顾可成为后续 Growth Score 的证据，也可由系统提出 Memory 建议，但 Memory 仍需单独确认。

该流程可概括为：

**用户输入 → FastAPI 身份与安全校验 → 读取已授权上下文 → OpenAI 分析与追问 → 用户补充和纠正 → OpenAI 生成报告草稿 → 后端校验 → 用户确认行动 → 用户主动保存 → Supabase 持久化 Reflection。**

## 8. 接口与通信原则

- 浏览器与 FastAPI 之间使用版本化的 HTTPS API。
- 普通资料读写使用请求—响应模式；AI 对话可以使用流式响应改善等待体验，但最终有效消息以服务端完成校验并保存的版本为准。
- 每个写操作都需要用户身份、资源归属校验和明确的业务状态前置条件。
- 对可能重复提交的关键操作，如保存 Reflection、确认 Memory 和生成阶段报告，应支持幂等处理，避免网络重试产生重复数据。
- 错误响应区分身份失效、权限拒绝、业务状态冲突、AI 暂时不可用和内容安全分支，前端据此给出可理解的反馈。
- OpenAI API 密钥、Supabase 服务端密钥和内部提示模板只保存在后端安全配置中。

## 9. 可靠性、性能与可观测性

### 9.1 可靠性

- OpenAI API 超时或不可用时，保留用户已输入内容和 Conversation 进度，允许稍后重试，不生成不完整的正式报告。
- AI 返回格式不合格时，由后端进行有限重试或要求重新生成；仍失败则明确提示，不把无效结果写为正式数据。
- 关键写入使用事务和幂等标识，避免重复 Reflection 或半完成关联。
- 数据库备份、恢复和迁移需要可验证，尤其保护档案、成长记录和用户确认状态。

### 9.2 性能与成本

- 只向 OpenAI 发送与当前阶段相关的上下文，历史对话优先使用经过确认的结构化摘要，控制延迟和 token 成本。
- 档案、历史列表等常规查询使用分页和必要索引。
- 报告生成等较慢操作在前端展示明确进度；随着规模增长，可迁移为后台任务。
- 记录每类 AI 任务的耗时、失败率和 token 使用量，但不在指标系统中保存敏感正文。

### 9.3 可观测性

系统应观察 API 成功率与延迟、数据库错误、AI 请求延迟与失败、结构校验失败、安全分支数量和关键业务漏斗。日志使用请求标识、用户匿名标识和业务实体标识串联链路，默认不记录完整用户输入、完整提示或 AI 敏感输出。

## 10. 隐私与安全架构

- Supabase Auth 负责身份会话，FastAPI 对每次请求重新验证身份和数据归属。
- 数据库使用行级访问控制，用户只能访问自己的数据。
- 浏览器只持有公开客户端配置与用户会话，不持有服务端密钥和 OpenAI API 密钥。
- 正式 Reflection、完整 Conversation 和 Memory 是三种独立保存选择。
- 敏感内容遵循最少收集、最少发送、最少保留；日志和监控对正文进行排除或脱敏。
- 用户可以查看、纠正、导出和删除个人数据；账号删除应协调清理 Profile、Conversation、Reflection、Growth Score、Memory 及相关来源关系。
- 数据导出和删除属于异步可能性较高的操作时，需要提供明确状态和最终结果。
- AI 生成内容保留模型和提示版本，用户确认内容保留确认时间，支持争议追溯和后续质量改进。

## 11. 未来扩展性

### 11.1 应用扩展

第一版的模块化单体应按身份、档案、对话、AI 编排、报告、成长记录、记忆和成长评估划分边界。用户量和任务量增长后，可优先拆分 AI 编排与报告生成，因为它们具有独立的延迟、成本和扩容特征；身份和核心数据业务可继续保持稳定。

### 11.2 异步任务

阶段性成长报告、批量数据导出、账号删除和历史数据复核未来可通过任务队列与后台工作进程处理。任务必须携带用户归属和幂等标识，并能查询状态、重试和审计。

### 11.3 AI 能力演进

AI 编排层通过统一适配接口隔离具体模型。未来可以按任务选择不同 OpenAI 模型、增加模型评估、提示版本实验、质量回归测试和成本路由，而不改变前端或核心数据模型。多模型能力属于内部演进，不需要在 MVP 向用户暴露参数选择。

### 11.4 数据检索与长期记忆

MVP 先使用结构化查询选择 Profile、Memory 和近期 Reflection。只有当历史数据规模证明需要时，再增加语义检索或向量索引。即使采用语义检索，也必须先通过用户权限、记忆开关、适用范围、有效期和来源状态过滤，不能把“检索到”误当作“允许使用”。

### 11.5 数据分析与产品改进

未来可以建立去标识化的产品分析数据层，观察建档完成率、五层对话完成率、报告确认率、Reflection 保存率和后续回顾率。分析事件与敏感成长正文应分离，默认不把用户内容用于与产品体验无关的分析。

### 11.6 多端与国际化

后端 API 与领域模型保持独立，可在未来支持移动端或其他客户端。界面文本、时区和语言在数据模型中保留扩展点，但 V1.0 不扩大为多语言和多端同步项目。

### 11.7 数据规模与合规演进

随着用户量、地区和合规要求变化，可进一步增加数据分区、区域化存储、密钥轮换、细粒度审计、保留策略自动化和安全审查。所有演进均应保持用户对 Profile、Reflection、Conversation 和 Memory 的可见与可控。

## 12. 第一版架构边界

V1.0 不引入微服务、复杂事件总线、知识图谱、多模型用户配置、高级推荐系统或完整成长地图可视化。第一版架构的成功标准是：

1. 用户可以安全地建立并确认个人档案。
2. 一次结构化 AI 对话可以中断、恢复并正确推进状态。
3. AI 输出经过约束、校验和用户确认，不把推测自动写成事实。
4. 用户可以主动将确认后的分析保存为可追溯的 Reflection。
5. 后续回顾、长期记忆和成长报告均使用经过授权且有来源的数据。
6. 系统可以在不推翻核心设计的前提下扩容、替换 AI 模型并拆分高负载模块。

该架构以 React、FastAPI、Supabase 和 OpenAI API 组成清晰的职责链：React 提供体验，FastAPI 执行业务与安全规则，Supabase 保存身份和成长数据，OpenAI 提供受控的分析与生成能力。最终的数据决定权始终保留在用户和 Growth Atlas 的业务规则中，而不是交给模型自动决定。
