# Growth Atlas Master Development Plan V1.0

> 版本：1.0
> 制定日期：2026-07-20
> 规划范围：从当前 V2 迁移状态到第一个可用 Beta 版本
> 计划口径：以一名 Founder + AI 工程执行为基准；时间是有效开发日，不含 Founder 等待、第三方审核和不可控故障时间。

## 0. 管理原则与 Beta 判定

本计划的目标不是“完成更多页面”，而是交付一条真实用户可以安全走完、数据可以追溯、失败可以诊断的决策闭环。

Beta 只有同时满足以下条件才可发布：

1. 新用户可以注册或登录，理解产品与数据边界，并建立、查看、修改个人档案。
2. 用户可以创建一个真实决策事件，补充事实、目标、约束和选项，获得结构化 AI 草稿。
3. AI 草稿明确区分事实、用户判断、假设、预测和未知信息；用户可以审阅，而不是被系统直接替代决定。
4. 用户可以记录自己的最终决定，创建行动计划，更新行动状态，并提交至少一次结果反馈。
5. 用户可以在历史页重新打开完整决策链；系统只使用经用户确认且与当前问题相关的长期信息。
6. 用户可以修改关键资料，并能删除自己的单个决策事件；隐私说明、危险情境提示和专业边界可见。
7. 核心流程在目标部署环境完成端到端验证；数据库 RLS、迁移、错误恢复、日志和最小发布回滚流程通过检查。
8. P0 缺陷为 0；不存在数据串号、密钥泄露、静默丢数据、AI 草稿被误标为用户决定等问题。

## 一、当前项目状态

### 1.1 已完成

#### 产品与知识层

- V2 产品方向已经明确：从泛化“个人成长助手”转为“AI 个人决策系统”。
- 核心闭环已定义为：记录问题、整理事实、明确目标与约束、比较选项、确认选择、记录行动、跟踪结果、沉淀决策认知。
- 已明确 AI 不替用户决定、不做伪精确人格评分、长期记忆必须有用户确认。
- 已有产品定位、AI 行为、架构、数据模型和知识库资料；仓库已有贡献规则与统一验证入口。

#### 数据库与领域模型

- V1 基础数据库、用户档案和 RLS 基础已经存在。
- V2 九类核心实体已建立：个人决策档案、档案版本、决策事件、AI 报告草稿、行动计划、行动项、反馈、记忆候选和决策记忆。
- V2 数据表采用 `user_id` 所有权、复合外键、生命周期枚举和确认时间约束。
- 已增加决策背景、AI 草稿关联、用户最终确认、行动计划和反馈的增量迁移。
- V1 表仍保留，未进行危险的自动回填或破坏性迁移。

#### 后端

- FastAPI、Supabase 身份令牌边界、Kimi/OpenAI 兼容客户端已经接入。
- 已有档案、V1 反思、报告、V2 决策事件、AI 草稿、用户确认、行动计划和反馈接口骨架。
- AI 草稿与用户最终决定在模型和持久化层面分离。
- 已有 `/api/health` 与不泄露密钥的 `/api/ready`。
- 当前后端 lint 通过，自动测试共 41 项通过。

#### 前端

- 已有 Supabase 登录入口、档案建立/确认页面。
- 已有 V2 决策创建、事件详情、AI 草稿审阅、决定确认、行动执行和反馈页面。
- 使用 session storage 保留当前决策页面位置。
- TypeScript 检查与生产构建通过。

### 1.2 未完成

- 当前 V2 工作仍处于大量未提交文件状态，尚未形成可回滚、可审查的稳定基线。
- 迁移主要经过静态与单元测试，尚未证明能在一个干净 Supabase 项目按顺序完整应用并通过 RLS 实测。
- V2 前端虽然已有页面，但尚未完成真实浏览器端到端验收、刷新恢复、空状态、重复提交、网络失败和移动端检查。
- 新旧 V1/V2 路径并存；`reflection`、mock growth report 和旧 timeline 容易造成用户入口与数据真源混乱。
- V2 的“事实/目标/约束/选项”输入仍偏自由文本，结构化程度和 AI 输入质量尚未完成产品验证。
- 行动与反馈已经有接口骨架，但提醒、到期状态、反馈触发条件和完整历史展示尚未形成产品闭环。
- 记忆候选与决策记忆只有领域/数据库基础，缺少用户审核界面、写入策略、撤销和使用可见性。
- V2 Growth Map/决策轨迹摘要尚未实现；现有 Growth Report 属于 V1 mock，不可作为 Beta 能力。
- 缺少前端自动测试、真实后端集成测试、RLS 测试、迁移回放测试和 Beta 级端到端测试。
- 缺少结构化运行日志、请求关联 ID、AI 请求耗时/错误指标、部署与回滚演练。
- 数据导出、账号级删除可以延后，但单事件删除、资料修正、隐私说明和数据使用透明度必须补齐。
- 旧的 MVP 文档仍描述 V1 产品，需要建立 V2 Beta 规格真源，避免产品与工程继续分叉。

### 1.3 最大风险

1. **工作基线风险：** 当前大量 V2 改动未提交，任何继续开发都可能让回归定位和回滚变得困难。
2. **假闭环风险：** 页面和接口“看起来齐全”，但没有在真实 Supabase、真实登录和真实 AI 请求下跑通完整闭环。
3. **双轨产品风险：** V1 reflection/growth report 与 V2 decision episode 同时存在，可能让用户和开发者都不知道哪个是真入口、哪个是真数据。
4. **长期记忆风险：** 如果为了差异化过早自动写入推断，可能污染用户画像并违背“用户确认”原则。
5. **AI 质量风险：** 只有 schema 正确不等于建议有用、安全或忠于用户事实；目前缺少固定评测集。
6. **隐私与权限风险：** RLS 设计存在不代表真实部署正确；必须用两个真实测试用户验证跨用户隔离。
7. **范围膨胀风险：** Growth Map、记忆、提醒和高级分析都容易扩张，挤压第一条可靠闭环的完成时间。

## 二、最终 MVP 定义

### 2.1 Beta 必须包含

#### A. 身份、边界与个人档案

- 邮箱注册、登录、登出和会话恢复。
- 首次使用时展示产品边界、AI 使用说明和数据保存说明。
- 用户建立、确认、查看和修改个人档案：当前阶段、目标、价值排序、现实约束和用户自述。
- AI 仅使用与当前决策相关的已确认档案信息。

#### B. 一次完整的决策事件

- 创建、保存草稿、继续编辑、删除未确认事件。
- 至少收集：决策问题、背景、目标、已知事实、关键未知、约束、候选选项。
- 用户可以从历史列表重新进入事件。
- 明确的生命周期：草稿、准备分析、AI 草稿、用户已决定、执行中、已复盘/归档。

#### C. AI 决策草稿

- 生成结构化草稿：目标、事实与缺口、约束、选项比较、盲点、建议及适用条件、行动建议。
- 明确标注事实/判断/假设/未知；不得虚构未提供的事实。
- AI 失败时保留用户输入，给出可重试错误，不产生半成品确认状态。
- 保存模型、prompt 版本、上下文快照摘要和生成时间，支持追溯。

#### D. 用户审阅与最终决定

- 用户能查看 AI 草稿，并记录接受、调整或不采用其建议。
- 最终决定必须由用户填写和确认；AI 草稿不能自动变成最终决定。
- 确认后保留所依据的草稿版本和用户自己的理由。

#### E. 行动、反馈与最小长期价值

- 用户创建并确认一个行动计划，包含行动项、完成标准、障碍/备用方案和复盘时间。
- 用户可以更新行动项状态。
- 到达复盘节点后可以提交结果、偏差、经验和下一步。
- 历史详情能串联“问题—AI 草稿—用户决定—行动—反馈”。
- Beta Growth Map 只提供基于真实记录的决策轨迹与少量证据摘要，不提供人格分数或确定性标签。

#### F. 安全、隐私与可靠性

- 高风险专业问题和紧急危险情境有明确提示，不把产品包装为专业替代品。
- 用户只能访问自己的数据；跨用户访问在 API 与数据库两层均被阻止。
- 单事件删除、档案修正可用；密钥只存在服务端。
- 核心错误可观测，数据库迁移可重复部署，Beta 有备份和回滚步骤。

### 2.2 Beta 明确不包含

- 社区、好友、排行榜、导师市场和多人协作。
- 自动代替用户报名、购买、发消息或执行外部决定。
- 语音、视频、数字人、多语言和原生移动 App。
- 复杂提醒系统、日历同步和完整任务管理。
- 人格测试、成长评分、决策能力分数和预测成功率。
- 未经用户确认的自动长期记忆。
- 大规模 V1 数据自动回填；Beta 期间优先保持兼容与可读，不冒险强迁移。
- 法律、医疗、投资等专业结论服务。

### 2.3 Beta 成功指标

产品验证指标：

- 至少 10 名目标用户完成首次建档，其中至少 7 名完成一次“创建—草稿—确认”流程。
- 至少 5 名用户创建行动计划，至少 3 名在约定时间提交反馈。
- 完成用户中至少 60% 认为系统帮助其看清了一个此前忽略的事实、约束或权衡。
- 严重误导、安全边界突破或跨用户数据事件为 0。

工程指标：

- 核心端到端流程通过率 100%，P0 缺陷为 0，P1 缺陷有明确处置结论。
- 目标环境的 API 5xx 率低于 1%；AI 失败不会丢失用户输入。
- 两用户 RLS 隔离测试、干净数据库迁移测试和回滚演练全部通过。
- 所有生产发布均可关联到版本、迁移清单和验证记录。

## 三、完整 Sprint 规划

### Sprint B0：冻结并验证 V2 当前基线

1. **Sprint 目标：** 将当前未提交的 V2 迁移成果整理为一个可审查、可回滚、测试通过的稳定基线。
2. **用户价值：** 降低后续开发破坏现有决策流程的概率。
3. **技术目标：** 核对迁移顺序、API 契约、领域状态机、环境模板和统一检查；清除阻塞性 lint/测试问题。
4. **修改范围：** 仅整理现有 V2 文件、文档、测试与配置；不新增产品功能。
5. **数据库变化：** 不新增业务表；在干净 Supabase 环境回放全部现有迁移并保存验证结果。
6. **API 变化：** 冻结现有 V2 endpoint 命名和响应结构；修复明显契约冲突。
7. **前端变化：** 确认所有 V2 页面可构建、入口可达；不做视觉扩展。
8. **测试标准：** 后端 lint/单测、前端构建、迁移回放、两个用户 RLS 冒烟测试全部通过；形成基线提交。
9. **完成时间估计：** 2—3 个开发日。

### Sprint B1：Beta 规格真源与主导航收敛

1. **Sprint 目标：** 明确 V2 Beta 唯一产品路径，停止 V1/V2 双入口继续扩散。
2. **用户价值：** 用户进入产品后知道下一步做什么，不会在“反思”和“决策”两个概念间迷路。
3. **技术目标：** 建立 V2 Beta 规格、路由/页面状态图、事件生命周期和 API 对照表。
4. **修改范围：** 产品文档、App 导航、V1 兼容入口标记、空状态和页面文案。
5. **数据库变化：** 无。
6. **API 变化：** 无新增接口；标记 V1 接口为 legacy，定义停写时间但暂不删除。
7. **前端变化：** 以“个人档案—新建决策—历史决策”为主导航；隐藏 Beta 不使用的 mock Growth Report 主入口。
8. **测试标准：** 登录后所有主入口可达；刷新和返回行为一致；不存在通往 mock 数据的主流程链接。
9. **完成时间估计：** 2—3 个开发日。

### Sprint B2：身份、档案与隐私边界加固

1. **Sprint 目标：** 让首次使用、档案确认、会话恢复和数据说明达到真实用户可用标准。
2. **用户价值：** 用户知道系统保存什么、AI 使用什么，并能修正自己的长期背景。
3. **技术目标：** 完成身份异常处理、档案状态校验、最小隐私同意记录和会话恢复。
4. **修改范围：** Auth、Profile、隐私/安全文案、权限错误处理。
5. **数据库变化：** 如合规记录确有需要，新增轻量 consent/version 字段或事件表；不得将同意信息塞入自由文本。
6. **API 变化：** 档案读取/更新/确认契约固化；必要时新增当前用户数据概览接口。
7. **前端变化：** 首次边界确认、档案查看/编辑、登出、过期会话提示、加载和错误状态。
8. **测试标准：** 未登录访问被拒绝；用户 A 无法读取/修改用户 B 数据；档案未确认时不能进入 AI 分析；敏感密钥不出现在浏览器包或响应中。
9. **完成时间估计：** 3—4 个开发日。

### Sprint B3：结构化决策输入与草稿恢复

1. **Sprint 目标：** 将自由文本困惑转化为足够完整、可保存、可恢复的决策输入。
2. **用户价值：** 用户不必一次写完，也能逐步看清目标、事实、约束、未知和选项。
3. **技术目标：** 固化输入 schema、自动保存策略、合法状态转换和重复提交保护。
4. **修改范围：** DecisionCreation、DecisionEpisodeDetail、V2 schema/service、错误提示。
5. **数据库变化：** 优先使用现有结构；只有现有 JSON/字段无法表达验收数据时才新增迁移，并记录 schema version。
6. **API 变化：** 创建、更新、读取、标记 ready、删除草稿接口完成幂等和冲突语义；返回稳定错误码。
7. **前端变化：** 分区输入目标/事实/未知/约束/选项；保存状态、字数限制、离开保护、刷新恢复和删除确认。
8. **测试标准：** 空输入、超长输入、重复点击、断网重试、刷新恢复、非法状态跳转和已确认事件不可编辑均有测试。
9. **完成时间估计：** 4—5 个开发日。

### Sprint B4：AI 草稿质量、安全与可追溯性

1. **Sprint 目标：** 让 AI 输出从“schema 可解析”提升为“真实可审阅的决策草稿”。
2. **用户价值：** 用户获得与个人情境相关、事实边界清楚、能揭示盲点的分析，而不是泛化建议。
3. **技术目标：** 版本化 prompt、精确上下文选择、结构化验证、超时/重试、失败恢复和质量评测集。
4. **修改范围：** Context Builder、AI service、draft service、prompt 契约、错误日志和评测样例。
5. **数据库变化：** 确保草稿保存模型名、prompt 版本、context snapshot、生成状态和错误摘要；缺失部分通过增量迁移添加。
6. **API 变化：** 生成草稿、查询生成状态、读取版本和重试接口契约；避免一次请求造成重复草稿。
7. **前端变化：** 生成进度、超时/失败重试、草稿版本提示、事实/假设/未知的清晰呈现。
8. **测试标准：** 至少 20 个固定决策案例评测；schema 通过率 100%；事实虚构、越权决定、高风险边界分别有反例测试；AI 失败不改变用户事件的可编辑数据。
9. **完成时间估计：** 5—6 个开发日。

### Sprint B5：用户审阅、最终决定与审计链

1. **Sprint 目标：** 完成 AI 草稿到用户自主决定的明确确认门。
2. **用户价值：** 用户保留最终决定权，并能看见“AI 建议”和“我的决定”之间的差异。
3. **技术目标：** 原子确认、草稿版本锁定、并发冲突处理和不可篡改的关键时间点。
4. **修改范围：** DecisionReview、确认服务、事件详情和状态机。
5. **数据库变化：** 使用现有 `confirmed_from_draft_id` 与确认约束；仅在审计证据不足时新增迁移。
6. **API 变化：** 确认接口必须要求 draft id、用户最终决定和理由；重复确认返回稳定结果或明确冲突。
7. **前端变化：** 接受/调整/不采用 AI 建议；填写最终决定、理由和确认提示；确认后只读展示。
8. **测试标准：** AI 草稿不能直接写入最终决定；跨用户、过期草稿、重复确认和并发确认均被阻止；确认链可重新读取。
9. **完成时间估计：** 3—4 个开发日。

### Sprint B6：行动计划、执行与反馈闭环

1. **Sprint 目标：** 让用户从“想清楚”走到“做了什么、结果怎样”。
2. **用户价值：** 决策形成具体下一步，并在真实结果中获得复盘价值。
3. **技术目标：** 行动计划确认、行动项状态、复盘时间、反馈提交和事件阶段更新保持一致。
4. **修改范围：** DecisionExecution、action/feedback API、服务与事件详情时间线。
5. **数据库变化：** 使用现有事务函数；必要时补充 `review_due_at`、反馈类型或唯一性约束，禁止隐式生成长期记忆。
6. **API 变化：** 创建/读取行动计划、更新行动项、提交/读取反馈；明确重复反馈和状态冲突规则。
7. **前端变化：** 行动清单、完成标准、障碍和备用方案、复盘日期、结果/偏差/经验表单、完整事件时间线。
8. **测试标准：** 未确认决定不能创建执行计划；行动更新和反馈用户隔离；重复请求不产生重复数据；完整闭环端到端测试通过。
9. **完成时间估计：** 4—5 个开发日。

### Sprint B7：确认式记忆与最小 Growth Map

1. **Sprint 目标：** 交付 Growth Atlas 相比通用聊天工具的最小长期价值，但不做自动人格判断。
2. **用户价值：** 用户能看到过去如何选择、发生了什么，并自主决定哪些经验可用于未来。
3. **技术目标：** 从已确认反馈提出可审阅的记忆候选；确认后才进入决策记忆；上下文使用可追溯、可撤销。
4. **修改范围：** memory service/API、用户审核界面、历史决策轨迹、Context Builder。
5. **数据库变化：** 优先使用现有 `memory_candidates` 与 `decision_memories`；补充必要的来源、撤销和使用记录字段。
6. **API 变化：** 列出/确认/拒绝记忆候选，查看/停用已确认记忆，读取最小决策轨迹摘要。
7. **前端变化：** “建议记住的经验”审核卡片、已确认记忆管理、基于证据的决策历史摘要；不显示评分。
8. **测试标准：** 未确认候选永不进入 AI 上下文；拒绝/停用后不再使用；每条摘要可回到原决策证据；数据不足时明确显示不足。
9. **完成时间估计：** 4—5 个开发日。

### Sprint B8：Beta 可靠性、部署与封闭测试

1. **Sprint 目标：** 把功能闭环转化为可安全邀请真实用户的 Beta 产品。
2. **用户价值：** 用户在常见设备和网络波动下仍能完成流程，遇到错误知道如何恢复。
3. **技术目标：** 端到端测试、RLS 验证、结构化日志、请求关联、部署/备份/回滚、性能和可访问性检查。
4. **修改范围：** 全栈缺陷修复、监控、部署配置、运行手册、Beta 反馈通道；冻结新功能。
5. **数据库变化：** 只允许缺陷修复迁移；在 staging 回放并备份，形成生产迁移清单。
6. **API 变化：** 冻结 Beta API；统一错误响应；补充必要的诊断信息但不暴露隐私。
7. **前端变化：** 全局错误页、加载/空状态、移动端、键盘操作、基础无障碍、隐私与反馈入口。
8. **测试标准：** 核心 E2E、双用户隔离、迁移回放、备份恢复、AI 超时、网络中断、移动端和主流浏览器验收；P0=0；Founder 完成最终验收脚本。
9. **完成时间估计：** 5—7 个开发日，加 5—7 个自然日封闭 Beta 观察。

## 四、开发优先级

### P0：没有它就不能发布 Beta

- 稳定基线、迁移回放、RLS 隔离和可回滚发布。
- 唯一 V2 主导航与完整决策闭环。
- 身份、档案确认、决策事件保存与恢复。
- AI 草稿的事实边界、安全边界、错误恢复和可追溯性。
- 用户最终决定确认门；AI 不能越权确认。
- 行动计划、行动状态、至少一次反馈和完整历史读取。
- 单事件删除、档案修正、隐私说明和密钥隔离。
- 核心 E2E、P0 缺陷清零、结构化运行日志。

### P1：强烈建议随 Beta 交付，但可在时间受限时缩小

- 结构化输入引导和更完整的未知/盲点提示。
- 确认式记忆候选与最小 Growth Map。
- AI 固定评测集扩展、耗时/错误指标和更好的重试体验。
- 移动端精修、基础无障碍和 Beta 内反馈入口。
- 事件归档、过滤和简单搜索。

### P2：Beta 后再评估

- 日历/第三方提醒、邮件通知和复杂任务管理。
- 高级 Growth Map 可视化、跨决策模式分析和个性化推荐。
- V1 数据批量回填、账号级完整导出和复杂删除编排。
- 多模型路由、语音、多语言、原生 App。
- 社交、导师服务、组织后台和自动外部执行。

## 五、Founder 参与节点

### 5.1 必须由 Founder 确认

| 决策节点 | 需要确认的内容 | 最晚时间 |
| --- | --- | --- |
| Gate F1：Beta 产品边界 | Beta 必须包含/不包含、目标用户、成功指标 | Sprint B1 结束前 |
| Gate F2：首次体验 | 首屏承诺、隐私说明、档案问题是否符合用户感受 | Sprint B2 结束前 |
| Gate F3：决策输入 | 哪些信息是“足够分析”的最低要求，交互是否过重 | Sprint B3 结束前 |
| Gate F4：AI 品质 | 20 个基准案例中建议的有用性、语气、挑战强度和安全边界 | Sprint B4 结束前 |
| Gate F5：用户确认 | 接受/调整/不采用 AI 建议的产品表达与最终确认文案 | Sprint B5 结束前 |
| Gate F6：反馈闭环 | 什么结果值得复盘、何时提醒、用户负担是否合理 | Sprint B6 结束前 |
| Gate F7：长期记忆 | 哪些内容值得长期保存，候选与确认界面是否建立信任 | Sprint B7 结束前 |
| Gate F8：Beta 发布 | 目标测试用户、隐私文本、已知限制、P1 缺陷接受清单 | 发布前 |

Founder 的确认应针对产品取舍、用户体验、风险接受和内容品味，不要求逐行审核代码。

### 5.2 可由工程负责人自主执行

- 在已批准产品边界内进行代码组织、命名、类型、测试和小范围重构。
- 修复测试、lint、构建、错误处理、幂等性、日志、性能和无障碍问题。
- 设计不改变产品语义的 API 细节和内部模块边界。
- 添加增量数据库约束、索引和安全修复，但涉及删除、不可逆迁移或用户数据语义改变时必须升级给 Founder。
- 维护文档真源、运行手册、评测集和发布检查清单。
- P0/P1 缺陷分级与修复；任何可能改变用户承诺的修复需回到 Founder Gate。

### 5.3 自动升级给 Founder 的条件

- 需要删除或不可逆转换用户数据。
- 需要扩大收集的个人或敏感信息范围。
- AI 要开始自动写长期记忆、替用户确认决定或执行外部动作。
- 需要改变目标用户、核心闭环、收费承诺或安全边界。
- 为按期发布必须放弃任何 P0 验收条件。
- 发现真实跨用户访问、密钥泄露、严重误导或安全事件。

## 六、风险管理

### 6.1 技术风险

| 风险 | 早期信号 | 预防与缓解 | 发布门槛 |
| --- | --- | --- | --- |
| 迁移在真实 Supabase 失败 | 本地 schema 与远端不一致 | 每个 Sprint 在干净 staging 回放；只新增迁移 | 全量回放通过 |
| RLS/鉴权串号 | service 测试只 mock HTTP | 两个真实用户做数据库与 API 隔离测试 | 跨用户访问全部拒绝 |
| AI 超时或无效 JSON | 重试后仍出现空白页 | 保存输入、幂等生成、超时与结构验证、可重试状态 | 不丢输入、不产生假确认 |
| 重复提交产生重复数据 | 双击/刷新后多条记录 | 请求幂等键、唯一约束或事务函数 | 核心写入重复测试通过 |
| 前端状态只靠 session storage | 换设备/直接链接无法恢复 | 以服务端事件状态为真源，URL/路由可恢复 | 刷新和直接进入通过 |
| 依赖/运行环境漂移 | `.venv` 或 lockfile 不可复现 | 固定 lockfile、统一检查、部署镜像/版本记录 | 干净环境构建通过 |

### 6.2 架构风险

| 风险 | 早期信号 | 预防与缓解 |
| --- | --- | --- |
| V1/V2 双写与双真源 | 同一决策出现在不同表且含义不同 | Beta 主流程只写 V2；V1 只读兼容并标记退出计划 |
| 状态机只存在于前端 | 可跳过确认直接执行 | 后端域规则 + 数据库约束 + API 状态冲突测试 |
| AI 输出与用户事实混合 | 草稿内容直接回写档案/决定 | AI 草稿、用户决定、记忆候选、确认记忆物理分层 |
| 过早抽象或多 agent 化 | 框架多于业务规则 | 单服务、简单接口优先；只有固定评测证明收益才升级 |
| 文档与代码分叉 | Sprint 名称/字段在多处不同 | 一个现行规格真源，其他文档链接而非复制 |
| Growth Map 变成伪人格评分 | 少量记录就显示稳定标签 | 只展示证据与不确定性；用户可否定和撤销 |

### 6.3 产品偏离风险

| 风险 | 控制方法 |
| --- | --- |
| 重新滑向泛化聊天或心理安慰 | 每个功能必须直接服务一个真实决策阶段；基准案例检查挑战与行动价值 |
| 变成“AI 替你决定” | 用户最终确认门不可绕过；文案与数据模型持续分离建议和决定 |
| 为长期价值过度收集数据 | 最小收集、用途可见、记忆需确认、支持修正/停用/删除 |
| 功能多但用户不复盘 | Beta 指标把行动计划和反馈完成率列为核心，不以页面数量衡量 |
| Founder 品味介入过晚 | 每个关键用户体验 Sprint 设 Gate，先用真实样例确认再扩展 |
| 为赶进度牺牲安全与可靠性 | P0 门槛不可用 P1/P2 功能交换；不满足则延迟发布 |

### 6.4 风险运行机制

- 每个 Sprint 开始时更新风险清单，结束时记录新增、关闭和接受的风险。
- 同类缺陷第二次出现时，必须升级为测试、约束、lint、迁移检查或运行手册。
- P0 缺陷立即停止相关功能扩展；先恢复数据与安全边界。
- AI 质量问题同时记录输入、上下文版本、prompt 版本、模型和输出类别，避免只凭印象修 prompt。
- 发布前进行一次独立的最终验证，不沿用开发过程中的假设。

## 七、最终路线图

### 7.1 时间线

| 阶段 | Sprint | 预计累计开发时间 | 关键里程碑 |
| --- | --- | ---: | --- |
| 基线稳定 | B0 | 第 1 周前半 | V2 当前成果可回滚、可验证 |
| 产品收敛 | B1 | 第 1 周后半 | 唯一 V2 主路径与 Beta 规格确认 |
| 可信入口 | B2 | 第 2 周 | 身份、档案、隐私边界可用 |
| 决策输入 | B3 | 第 3 周 | 可保存恢复的结构化事件 |
| AI 核心价值 | B4 | 第 4—5 周 | 草稿质量、安全与评测过门 |
| 用户自主权 | B5 | 第 5 周 | AI 建议到用户决定的确认链完成 |
| 行动闭环 | B6 | 第 6 周 | 决定、行动、结果串联完成 |
| 长期价值 | B7 | 第 7 周 | 确认式记忆与最小决策轨迹 |
| 发布加固 | B8 | 第 8—9 周 | E2E、部署、回滚与封闭 Beta |
| Beta Release | — | 约第 9—10 周 | Founder 签署 Gate F8，邀请首批用户 |

在无重大架构返工、第三方服务阻塞和 Founder Gate 长时间等待的情况下，预计 **8—10 周**到达可用 Beta。若资源受限，优先缩小 B7 的展示范围，不削减 B0、B2、B4、B5、B6、B8 的 P0 门槛。

### 7.2 每个 Sprint 的固定节奏

1. 开始：确认目标、范围、约束和可机器验证的完成条件。
2. 开发中：小步提交；数据库/API/前端各自通过局部检查。
3. 中间 Gate：关键用户体验先用一个真实案例演示，避免整 Sprint 做完才发现方向错误。
4. 结束：完整自动检查、真实环境冒烟、变更审查、风险更新和 Founder 必要确认。
5. 只有通过验收才进入下一 Sprint；未完成项重新分级，不以“代码已写”视为完成。

### 7.3 Beta Release 最终检查清单

- [ ] Beta 产品边界与已知限制由 Founder 确认。
- [ ] 所有数据库迁移在干净 staging 按顺序成功应用。
- [ ] 两个用户的 RLS/API 隔离测试通过。
- [ ] 注册/登录—档案—决策输入—AI 草稿—用户决定—行动—反馈—历史回看 E2E 通过。
- [ ] AI 固定评测达到已批准门槛，高风险案例通过。
- [ ] 断网、超时、重复提交、刷新和过期会话可以恢复。
- [ ] 单事件删除、档案修改、隐私说明可用。
- [ ] 生产密钥、CORS、日志隐私和 readiness 配置复核完成。
- [ ] 备份、迁移、部署、回滚和故障升级手册演练完成。
- [ ] P0 缺陷为 0，P1 接受项有负责人和处理日期。
- [ ] 首批 Beta 用户、反馈方式、观察周期和停止条件已确定。

当以上项目全部满足后，版本标记为 **Growth Atlas Beta 1**。Beta 期间优先验证用户是否完成行动与反馈闭环，不立即扩展 P2 功能。
